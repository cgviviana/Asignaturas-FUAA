Construcción paso a paso de la app MediStockYa

OVA en HTML para documentar el proceso completo: preparación del material, adaptación del prompt, verificación del flujo, edición dentro de Lovable y publicación final. Se apoya en las capturas reales del proyecto para que el recorrido sea claro y replicable.

Objetivo del OVA: dejar un recurso reutilizable para reconstruir o explicar cómo se levantó la app MediStockYa en Lovable sin perder de vista la maquetación, los créditos del plan gratuito y las correcciones funcionales hechas sobre el flujo.

1. Pasos previos

Antes de abrir Lovable conviene dejar listo el insumo base. Esta fase evita gastar créditos corrigiendo cosas que ya podían definirse desde el principio: objetivo, pantallas, paleta, flujo, imágenes y qué comportamiento debía tener el asistente IA.

Material mínimo

  • Nombre de la app y problema que resuelve.
  • Listado de pantallas esperadas.
  • Paleta, tipografía y estilo visual.
  • Textos reales en español, no lorem ipsum.
  • Sedes, medicamentos, alertas y reportes que se quieren simular.

Maquetación e imágenes

  • Elegir 4 a 6 pantallas clave para usar como referencia.
  • Si la maqueta está en PowerPoint, exportar diapositivas a PNG/JPG.
  • Guardar logo, íconos y banners por separado si ya existen.
  • Nombrar los archivos con sentido: inicio, login, reporte, perfil.

Si ya existe en Adalo

  • Tomar capturas de cada flujo importante.
  • Anotar colecciones, formularios y validaciones actuales.
  • Registrar botones, acciones y lógica de alertas.
  • Pasar a Lovable por referencia visual y funcional, no por importación directa.
  1. Definir el alcance. En este caso la app debía permitir registro, consulta de medicamentos, comparación, reporte, alertas, perfil y un bloque básico del asistente IA.
  2. Preparar la referencia visual. El mejor resultado se obtiene cuando Lovable recibe imágenes claras de la maquetación y un prompt que diga que esas imágenes son la referencia principal de estructura y estilo.
  3. Separar contenido de apariencia. La lógica de datos (usuarios, medicamentos, sedes, reportes, alertas) se pide en el prompt; el look & feel se guía con imágenes, colores y tipografía.
  4. Planear para el plan gratuito. Lo recomendable es un prompt maestro, luego una ronda de corrección funcional y después usar Visual edits para cambios finos.
Consejo práctico: si no tienes aún logo o imágenes finales, igual puedes construir la app desde cero con placeholders. Después reemplazas esos elementos desde Visual edits o con una nueva iteración cuando vuelvan los créditos.

2. Prompt

El prompt se adaptó para que Lovable entendiera tres cosas a la vez: la identidad visual de la maqueta, la arquitectura funcional de la app y la estructura de datos que debía quedar lista para futuro backend.

Qué debía decir el prompt maestro

  • Construir la app desde cero, no agregar una sección suelta.
  • Usar las imágenes adjuntas como referencia visual principal.
  • Trabajar en español y con textos realistas.
  • Definir todas las pantallas: bienvenida, registro, login, inicio, disponibilidad, comparador, reporte, visualización, alertas y perfil.
  • Incluir el bloque básico del asistente IA.
  • Preparar estructura de usuarios, medicamentos, sedes, alertas, reportes y consultas del asistente.

Dónde y cómo subir imágenes

  • Abrir un proyecto nuevo y usar el botón + del cuadro de prompt.
  • Elegir Attach y cargar las capturas más representativas.
  • Si la maqueta está en PPT, conviene exportar primero las diapositivas a PNG/JPG.
  • Si vienes de Adalo, las capturas del flujo sustituyen una importación directa.
Estrategia de créditos: con plan gratuito conviene evitar muchos prompts pequeños. Lo más rentable es hacer un prompt maestro, una pasada de corrección funcional y dejar lo demás para Visual edits.
Quiero que construyas desde cero una app web mobile-first llamada MediStockYa usando como referencia visual principal las imágenes adjuntas de mi maquetación. IMPORTANTE: - Usa las imágenes adjuntas como guía principal de estructura, estilo, distribución y jerarquía visual. - La app debe quedar en español. - No uses lorem ipsum. - Debe incluir registro, login, disponibilidad, comparador, reporte, alertas, perfil y un bloque básico del asistente IA. - Debe dejar lista la estructura para guardar usuarios, medicamentos, sedes, reportes, alertas y consultas del asistente. - La base de datos puede iniciar vacía; los registros se llenarán con el tiempo. - Si faltan logo o imágenes finales, usa placeholders fáciles de reemplazar después.

Idioma y tono

Se pidió toda la interfaz en español, con lenguaje claro para usuario adulto y senior. Esto era importante para que no aparecieran textos genéricos en inglés ni saludos impersonales como “Hola usuario”.

Correcciones después del prompt

La segunda ronda no reconstruyó la app: corrigió sedes visibles, personalización de alertas, saludo con nombre real, identidad del asistente (MediBot), carga de foto de perfil y subida de fórmula médica en reportes.

Resumen útil: el prompt debe combinar objetivo, diseño, pantallas, estructura de datos y restricciones. Las imágenes se encargan del parecido visual; el prompt se encarga de la lógica.

3. Verificación de flujo

Una vez Lovable arma la base, la revisión no se hace “por pantalla bonita” sino por flujo completo. El objetivo es detectar qué falta, qué se repite y qué no responde a la lógica funcional del caso.

  1. Inicio y bienvenida. Revisar si el saludo usa el nombre real, si los accesos principales están visibles y si el bloque del asistente IA se entiende.
  2. Disponibilidad. Confirmar que aparezcan sedes reales o simuladas con nombre y dirección, no solo placeholders genéricos.
  3. Comparador. Verificar orden por precio, disponibilidad y sede, y que las tarjetas distingan claramente cada criterio.
  4. Reporte. Comprobar el formulario y la carga de fórmula médica para que el asistente pueda orientar siguientes pasos.
  5. Alertas. Revisar que sean configurables por medicamento y sede, no alertas masivas sin personalización.
  6. Perfil. Confirmar foto, datos editables, acceso a alertas y salida de sesión.
Qué se corrigió durante esta verificación: listado de sedes, nombre del asistente, saludo con nombre, personalización de alertas, carga de foto de perfil, fórmula médica y opciones cuando el medicamento no se encuentra.
Punto de revisión Qué debía pasar Señal de alerta
Saludo y nombre Mostrar nombre real del registro o un saludo neutro. Mensajes como “Hola usuario”.
Sedes Listar puntos, direcciones y tipo de sede. No mostrar ninguna lista o repetir tarjetas sin sentido.
Alertas Elegir qué medicamento y qué sede seguir. Alertas generales para todo.
Asistente IA Tener identidad, utilidad y acciones claras. Chat genérico o poco confiable.
Reporte Permitir cargar fórmula y registrar detalle. Formulario simple sin contexto clínico.

4. Ediciones

Cuando se acaban los créditos no todo se detiene. Hay paneles y botones que siguen sirviendo para revisar, ordenar o pulir la app sin enviar un nuevo mensaje al asistente.

Botón o panel Para qué sirve ¿Conviene usarlo con 0 créditos?
Visual edits Editar texto, colores, espaciados, imágenes y detalles visuales. Sí En cambios normales suele ser la mejor opción.
Ask Lovable + flecha enviar Enviar un prompt nuevo o pedir una corrección por chat. No Consume créditos del workspace.
History Revisar versiones y cambios previos del proyecto. Sí Útil para comparar antes y después.
Knowledge Guardar contexto persistente para próximos prompts. Sí, pero no modifica la app por sí solo.
Attach Dejar listas imágenes o archivos para el próximo prompt. Sí, pero el efecto real llega al enviar el mensaje.
GitHub Sincronizar/exportar el proyecto para seguirlo fuera de Lovable. Sí Recomendado para respaldo.
Publish Publicar la versión actual como web app compartible. Sí Sirve para mostrar avance o demo.
Regla simple: si el botón abre configuración, historial, preview o edición visual, normalmente puedes usarlo; si dispara una instrucción nueva al chat, espera a que se renueven los créditos.

Qué panel usar para corregir faltantes

  • Visual edits: para textos, imágenes, tamaños, colores y ajustes rápidos.
  • History: para revisar lo que cambió entre una iteración y otra.
  • Knowledge: para dejar escritas las reglas del proyecto antes del siguiente prompt.
  • GitHub: para sacar el código o versionarlo fuera de Lovable.

Qué anotar para la siguiente ronda

  • Qué sedes faltan o sobran.
  • Qué textos siguen sonando genéricos.
  • Qué formularios requieren más campos.
  • Si la experiencia todavía se ve demasiado web y poco móvil.
  • Qué imagen o logo debe reemplazar cada placeholder.

5. Publicación

La publicación en Lovable convierte el proyecto actual en una web app en vivo con URL compartible. Para una demo funcional, este paso permite socializar el avance aunque todavía falten assets finales o backend real.

  1. Entrar al modal Publish. Elegir la URL generada o editarla si se desea un nombre más claro para el proyecto.
  2. Definir el acceso. En plan gratuito la opción típica es Public, es decir, cualquiera con el enlace puede abrir la web app publicada.
  3. Completar Website info. Cambiar título, descripción y, si ya está disponible, subir una imagen social para el enlace.
  4. Revisar el resumen. Confirmar URL, acceso y metadatos antes de pulsar Publish.
  5. Probar la app en vivo. Abrir la URL, compartirla y comprobar el flujo publicado.
Cierre del proceso: publicar no reemplaza la revisión. Cada ajuste importante debe volver a publicarse para que el enlace muestre la versión actualizada.
